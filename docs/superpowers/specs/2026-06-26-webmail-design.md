# WebMail System Design

## Background

This is a greenfield web system for collecting authorized buyer mailbox data, reviewing the authorization, and exposing approved buyer mail to assigned suppliers. The first version is a single web application with built-in background jobs.

The main business rule is privacy-scoped collection: the system only stores emails matching configured allowed sender rules. Non-matching mailbox content is not stored.

## Goals

- Let administrators generate random buyer card links in batches.
- Let buyers enter through a card link, authorize one Gmail account, and remove authorization only when business status allows it.
- Let sales users view and selectively delete their own buyers.
- Let administrators manage users, buyers, email review, supplier assignment, mail visibility, and sender rules.
- Let suppliers view only assigned buyers whose email authorization has been approved as normal.
- Trigger short-period mail refresh when a supplier opens a buyer mail page.
- Keep Outlook and other providers possible later through a provider abstraction.

## Non-Goals For Version 1

- No Outlook integration in version 1.
- No Gmail webhook or Google Pub/Sub integration in version 1.
- No collection or storage of emails from non-allowed senders.
- No sales access to mail body content.
- No multi-supplier collaboration workflow; one buyer is assigned to one supplier in version 1.
- No attachment file storage in version 1. The system stores attachment metadata only.

## Architecture

Use a single web application with background jobs:

- Web UI and API serve buyer entry, backend login, admin pages, sales pages, and supplier pages.
- Background jobs run mail synchronization and retry failed syncs.
- The database stores users, buyers, email accounts, mail snapshots, sender rules, supplier assignments, sync jobs, and audit logs.
- The email provider layer starts with Gmail and exposes a provider interface for future Outlook support.

The complexity is mainly in OAuth, data access boundaries, status transitions, and mail synchronization. A single application keeps these rules easier to enforce in the first version.

## Buyer Entry And Link Model

Administrators can generate a specified number of buyer links. Each link contains a random card number or token and may include a sales field such as `saleid=1`.

Example:

```text
/buyer/verify?card=<random-card>&saleid=1
```

Rules:

- The card number is generated by the system.
- The card number must be long, random, unique, and hard to guess.
- Buyer entry only validates that the card exists, is valid, and is not deleted or disabled.
- A buyer can authorize only one email account.
- `saleid` determines sales ownership when the link is generated.
- Sales ownership must be stored on the buyer or link record. Do not trust a modified `saleid` query parameter on later visits.

## Roles And Permissions

### Buyer

Buyers can:

- Open their own card link.
- View their own authorization status.
- Authorize Gmail.
- Remove authorization only when email status is `NotAuthorized`, `PendingReview`, or `Rejected`.

Buyers cannot:

- View mail content.
- View other buyers.
- Remove authorization when email status is `Normal` or `Abnormal`.

When removal is not allowed, show:

```text
正在处理中，无法删除
```

### Sales

Sales users can:

- View buyers belonging to their own `saleid`.
- View buyer card, email address if present, email review status, supplier processing status, and timestamps.
- Delete their own buyers only when email status is `NotAuthorized`, `PendingReview`, or `Rejected`.

Sales users cannot:

- View mail body content.
- Review email authorization.
- Delete buyers whose email status is `Normal` or `Abnormal`.
- View or modify buyers belonging to other sales users.

Sales deletion should be implemented as soft delete.

### Administrator

Administrators can:

- Generate buyer links in batches.
- Manage sales users, including disabling login and deleting or soft-deleting sales users.
- Delete buyers.
- View buyer mail snapshots.
- Review buyer email authorization as normal or rejected.
- Assign buyers to suppliers.
- Manage suppliers.
- Configure allowed sender rules.
- Configure initial sync range and sync frequency.

Administrator deletion of a sales user must not delete that user's associated buyers.

### Supplier

Suppliers can:

- View buyers assigned to them when email status is `Normal`.
- View stored mail snapshots for assigned normal buyers.
- Set supplier processing status to `Failed` or `Completed`.
- Trigger an active synchronization window by opening a buyer mail page.

Suppliers cannot:

- View unassigned buyers.
- View buyers whose email status is `NotAuthorized`, `PendingReview`, `Rejected`, or `Abnormal`.
- Delete buyers.
- Change email authorization review status.

## Status Model

### Card Status

- `Unused`
- `Entered`
- `Authorized`
- `DeletedOrDisabled`

### Email Authorization Status

- `NotAuthorized`: no active email authorization.
- `PendingReview`: Gmail OAuth succeeded and the email is waiting for administrator review.
- `Normal`: administrator approved the email account.
- `Rejected`: administrator rejected the email account.
- `Abnormal`: token refresh failed, permission was revoked, or Gmail API returned an authorization error.

Buyer removal of authorization is allowed only in:

- `NotAuthorized`
- `PendingReview`
- `Rejected`

Buyer removal of authorization is blocked in:

- `Normal`
- `Abnormal`

When a buyer removes authorization, the system deletes the OAuth token and active email binding. Previously synchronized matching mail snapshots remain available to administrators for audit by default, but the buyer is no longer visible to suppliers unless a new email is authorized, reviewed, and approved.

### Supplier Processing Status

- `Unprocessed`
- `Failed`
- `Completed`

Only suppliers assigned to the buyer, and administrators, can change supplier processing status.

## Data Model

Recommended tables:

| Table | Purpose |
| --- | --- |
| `users` | Backend users: administrators, sales users, suppliers |
| `roles` / `user_roles` | Role model; version 1 can simplify to `users.role` |
| `buyers` | Buyer record: card number, sales owner, email status, supplier status, soft delete state |
| `buyer_links` | Batch-generated card link records; can be merged into `buyers` in version 1 |
| `email_accounts` | Buyer mailbox authorization: provider, email address, encrypted token fields, sync state |
| `email_messages` | Stored snapshots for emails matching allowed sender rules |
| `allowed_senders` | Configured sender rules |
| `buyer_supplier_assignments` | Buyer-to-supplier assignment; one buyer to one supplier in version 1 |
| `sync_jobs` | Mail sync task records, status, failures, retries |
| `audit_logs` | Review, deletion, assignment, unlinking, supplier status changes, sync errors |

Constraints:

- `buyers.card_no` is unique.
- `buyers.sale_id` stores sales ownership.
- `email_accounts.buyer_id` is unique.
- `email_messages` deduplicates by `email_account_id + provider_message_id`.
- `allowed_senders` supports exact email matches in version 1.
- Domain rules such as `@example.com` can be added later.

## Gmail OAuth Flow

1. Buyer opens the card link.
2. The system validates that the card exists, is valid, and is not deleted or disabled.
3. Buyer selects Gmail.
4. The system redirects to Google OAuth and requests mail read permissions.
5. OAuth callback stores provider, email address, access token, refresh token, and provider user data.
6. Refresh token and sensitive provider data are encrypted at rest.
7. After successful authorization, email status becomes `PendingReview`.
8. Administrator reviews the authorization.
9. If approved, email status becomes `Normal`.
10. If rejected, email status becomes `Rejected`.

If the buyer already has an email account:

- Same provider and same email: update tokens and set status back to `PendingReview`.
- Different email: allow replacement only when the current status allows removal; otherwise show `正在处理中，无法删除`.

## Mail Synchronization

Synchronization only runs for buyers whose email status is `Normal`.

Rules:

- Query Gmail with configured allowed sender rules.
- Store only matching emails.
- Do not store non-matching emails.
- Initial sync pulls only a configured recent window, such as the last 30 days.
- Incremental sync records `last_sync_at` or Gmail `historyId`.
- Token refresh failure, revoked permission, or Gmail authorization errors set email status to `Abnormal`.
- Sync errors must be written to `sync_jobs` and `audit_logs`.

Stored message snapshot fields:

- Provider message id.
- Provider thread id.
- Sender.
- Recipients.
- Subject.
- Sent time.
- Text body snapshot.
- HTML body snapshot.
- Attachment metadata.

## Supplier Active Sync Window

When a supplier opens a buyer mail page:

1. The system verifies that the supplier is assigned to the buyer and the buyer email status is `Normal`.
2. The system creates or refreshes an active sync window for that buyer.
3. During the active window, background jobs sync that buyer more frequently, for example every 2 to 5 minutes.
4. The active window expires naturally, for example after 30 minutes.
5. A global low-frequency sync still scans normal accounts, for example every 30 to 60 minutes.

This avoids Gmail webhook complexity in version 1 while still keeping actively viewed mail relatively fresh.

## Pages

### Buyer Pages

- `/buyer/verify?card=...&saleid=...`: validates the card and starts the buyer flow.
- `/buyer/email`: shows authorization status and Gmail authorize or unlink actions.

Buyer-facing status text should cover:

- Not authorized.
- Pending review.
- Rejected.
- Processing, cannot delete.
- Authorization abnormal, contact administrator.

### Sales Backend

- Buyer list scoped to the logged-in sales user.
- Columns: card, email, email review status, supplier processing status, created time.
- Delete button only for `NotAuthorized`, `PendingReview`, and `Rejected`.

### Administrator Backend

- Sales management.
- Supplier management.
- Buyer management.
- Batch link generation.
- Buyer-to-sales assignment.
- Buyer-to-supplier assignment.
- Email review queue.
- Buyer mail viewer.
- Allowed sender configuration.
- Sync configuration.

### Supplier Backend

- Assigned normal buyer list.
- Buyer mail list.
- Buyer mail detail.
- Supplier processing status actions: failed and completed.

## API Modules

- `AuthController`: backend login, logout, current user.
- `BuyerEntryController`: card validation, buyer authorization status, buyer unlinking.
- `OAuthController`: Gmail authorization redirect and callback.
- `AdminController` or split admin modules: sales, suppliers, buyers, review, configuration.
- `SalesController`: sales-scoped buyer list and allowed buyer deletion.
- `SupplierController`: supplier-scoped buyers, mail, and processing status updates.
- `MailSyncJob`: background mail sync orchestration.
- `EmailProvider`: interface implemented by `GmailProvider`.

Every API must enforce permissions server-side.

## Security And Compliance Notes

- Store OAuth refresh tokens encrypted at rest.
- Do not return token values through any API.
- Use soft delete for buyers and sales where auditability matters.
- Audit all review decisions, buyer deletion, authorization removal, supplier assignment, supplier status changes, and sync failures.
- Keep mail access scoped to allowed sender rules.
- Do not expose mail bodies to sales users.
- Do not expose pending, rejected, abnormal, or unassigned buyers to suppliers.

## Testing Focus

- A buyer can bind only one email account.
- Buyer unlinking is allowed only for `NotAuthorized`, `PendingReview`, and `Rejected`.
- Buyer unlinking is blocked for `Normal` and `Abnormal`.
- Sales users can view only their own buyers.
- Sales users can delete only their own buyers in allowed statuses.
- Suppliers can view only assigned buyers with `Normal` email status.
- Administrators can review emails and make approved buyers visible to suppliers.
- Synchronization stores only messages matching allowed sender rules.
- Synchronization deduplicates provider messages.
- Token refresh or Gmail authorization errors set email status to `Abnormal`.
- Supplier mail page access creates or refreshes the active sync window.

## Open Decisions

- Exact backend framework and frontend framework are not selected yet.
- Exact database engine is not selected yet.
- Exact random card length and display format should be chosen during implementation planning.
- Whether administrators can permanently purge historical mail snapshots after buyer unlinking should be decided before production deployment.
